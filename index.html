Real Issues I Solved (Application Support Perspective)

Cloud API Configuration Troubleshooting:

AWS Bedrock · AI Resume Analyzer Project · Configuration Incident

Type: Production Incident
Severity: High (Feature unavailable)
Layer: API Configuration Environment

CARD 1 — PROBLEM

AI analysis feature blocked at runtime

The application threw "[bedrock] failed: The provided model identifier is invalid." when calling AWS Bedrock during resume analysis. AI-generated insights were completely unavailable, while the rule-based scoring layer continued to function normally — confirming the issue was isolated to the Bedrock integration layer.

Bedrock API error AI layer affected Fallback intact

CARD 2 — INVESTIGATION

Systematic config and runtime review

  1. Verified Bedrock client initialization — confirmed SDK was loading without errors
  2. Reviewed .env.local — modelId value appeared correct visually, no obvious errors
  3. Added runtime logging to capture region and modelId at request time
  4. Compared logged values against expected Bedrock model ID — mismatch detected
  5. Identified trailing whitespace in env variable as likely cause

CARD 3 — ROOT CAUSE

Config mismatch, not logic failure

The modelId passed to the Bedrock API contained a trailing whitespace character introduced when copying the model identifier into the environment variable.

No issues with IAM permissions, region config, or SDK version — the error was entirely in the identifier string.

CARD 4 — SOLUTION

Config correction + defensive coding

  1. Corrected the modelId value in .env.local
  2. Applied .trim() to all env variable reads to prevent recurrence
  3. Improved startup logging to print sanitized config values
  4. Restarted the app and confirmed Bedrock API returned successful responses
  5. Verified AI analysis output matched expected behavior end-to-end

CARD 5 — PRODUCTION VALIDATION & SERVICE VERIFICATION

Verification & regression validation

  • Validated Bedrock API returned successful responses after config fix
  • Tested multiple resume inputs to confirm stable AI output generation
  • Verified fallback (rule-based analysis) still works when AI is unavailable
  • Tested both local and production environments for consistency
  • Confirmed no regression in API routes or scoring logic

Ensured the issue was fully resolved and did not reappear across environments, demonstrating a QA-focused validation approach beyond simple bug fixing.

CARD 6 — KEY LESSONS

Many “AI bugs” are actually configuration bugs

This incident reinforced that silent failures at the cloud API layer often have nothing to do with model logic or application code. The real culprits are invisible characters in environment variables, mismatched service identifiers, and the absence of runtime observability.

Environment variables Cloud service identifiers Runtime validation Observability Defensive coding (.trim())
Why this matters for IT Support, Application Support & QA: this case demonstrates end-to-end troubleshooting across cloud configuration, environment variables, runtime validation, and service recovery — the same systematic debugging process required in real-world support and cloud operations roles.

Production Deployment Troubleshooting: DOMMatrix Runtime Error

Vercel Deployment · PDF Parsing Pipeline · Environment Mismatch Incident

Type: Production Incident
Severity: High (User-impacting)
Focus: Runtime Environment Dependency

CARD 1 — PROBLEM

PDF analysis broke only in production

The application worked correctly on localhost, but after deployment to Vercel, PDF parsing failed with "DOMMatrix is not defined". As a result, the resume analysis feature became unavailable in production, affecting all users.

Production bug Environment mismatch Localhost worked

CARD 2 — INVESTIGATION

Runtime-focused debugging across environments

  1. Cleared cache and redeployed — issue persisted
  2. Checked environment variables and deployment settings — no obvious errors
  3. Updated package versions to rule out dependency mismatch
  4. Compared local vs production behavior to confirm environment-specific failure
  5. Reviewed Node.js runtime differences and browser-only API dependencies

CARD 3 — ROOT CAUSE

Browser API dependency inside server runtime

The issue came from pdfjs-dist, which relied on browser-specific APIs such as DOMMatrix. These APIs were not available in the Node.js runtime used by the deployed environment.

This was not a logic bug in the feature itself — it was a runtime compatibility issue.

CARD 4 — SOLUTION

Library replacement + production validation

  1. Identified the incompatible PDF parsing dependency
  2. Replaced pdfjs-dist with a Node.js-compatible alternative
  3. Redeployed the application to Vercel
  4. Retested the upload and parsing flow in production
  5. Confirmed resume analysis was restored successfully

CARD 5 — PRODUCTION VALIDATION & SERVICE VERIFICATION

Production Verification & regression validation

  • Executed end-to-end testing for PDF upload, parsing, and resume analysis
  • Verified the fix specifically resolved the production-only failure
  • Confirmed expected behavior remained stable in localhost and Vercel
  • Validated the new parsing dependency was compatible with server runtime requirements
  • Checked for regression in adjacent upload and analysis workflows

This testing phase ensured the issue was not only fixed, but also validated against runtime compatibility risks and regression scenarios.

CARD 6 — KEY LESSONS

“Works on my machine” is not root cause analysis

This case reinforced a critical support principle: when an issue appears only after deployment, the real question is not whether the code works locally, but whether the runtime environment supports the same APIs, dependencies, and execution model.

Production debugging Runtime compatibility Dependency validation Deployment QA Systematic troubleshooting
Why this matters for IT Support, Application Support & QA: this case shows how production incidents often come from environment differences, not just code defects.
